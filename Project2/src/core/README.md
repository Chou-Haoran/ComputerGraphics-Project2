# core 模块

`core/` 放的是整个项目都在依赖的底层类型和工具。它不是“某个渲染特性”的实现，而是给所有模块打地基。

## 这个模块为什么重要

在路径追踪器里，几乎每个模块都会反复用到这些内容：

- 向量和点运算
- 射线
- 包围盒
- 公共常量
- 随机采样
- 进度条和小工具函数

如果这些基础能力分散在各处，后面会很难维护。因此这个模块的目标是：

- 尽量减少重复实现
- 给几何、灯光、材质、渲染器提供统一基础设施
- 避免到处拷贝数学代码

## 主要文件

- `Vector.hpp` / `Vector.cpp`：`Vector2f`、`Vector3f` 以及常用向量运算
- `Ray.hpp`：射线结构，保存 `origin` 和 `direction`
- `Bounds3.hpp`：AABB 包围盒，用于 BVH 和相交加速
- `global.hpp`：全局常量、随机数、辅助函数、进度显示

## 这些类型会被谁用到

- `geometry/`：大量使用 `Vector3f`、`Ray`、`Bounds3`
- `scene/`：积分时要用向量运算、随机采样、`EPSILON`
- `material/`：BSDF 评估和采样高度依赖向量与数学函数
- `light/`：灯光方向、PDF、可见性判断依赖底层数学
- `renderer/`：采样控制、进度条、像素循环会用到 `global.hpp`

## 常见工具说明

### `Vector3f`

这是项目里最常用的基础类型之一。它既表示：

- 几何坐标
- 法线
- 颜色
- 辐射亮度

使用时要自己清楚“这个三维量当前代表什么语义”。

### `Ray`

射线是路径追踪器的核心数据结构：

- 主光线由 `camera/` 生成
- 阴影射线与间接 bounce 射线由 `scene/` 生成
- 几何模块负责和它做相交测试

### `Bounds3`

主要服务于 BVH：

- 几何模块构建每个物体或三角形的包围盒
- `BVHAccel` 用它做层次划分和遍历剪枝

### `global.hpp`

这里面通常放：

- `EPSILON`
- `RussianRoulette` 相关小工具
- `clamp`
- 角度转换
- 随机数函数
- 进度条显示

## 使用时要注意什么

### 随机数

- `get_random_float()` 使用线程局部随机数生成器
- 这对 OpenMP 并行渲染很重要，否则不同线程会互相争用同一个随机源
- 如果你新增采样逻辑，优先沿用现有随机函数，不要自己随便再造一套全局 RNG

### `EPSILON`

`EPSILON` 在渲染器里不是“小装饰”，而是避免数值问题的关键：

- 阴影射线起点偏移
- 自相交抑制
- 分母接近 0 的保护
- 法线和方向退化判断

如果你发现某段代码在边缘情况不稳定，第一件事通常就是检查：

- 有没有偏移 ray origin
- 有没有对极小值做保护

### `clamp`

路径追踪里大量地方都要手动限制数值范围，例如：

- 颜色输出到 `[0, 1]`
- roughness 限制到合法范围
- `dot` 结果限制到 `[-1, 1]` 或 `[0, 1]`

统一使用现有工具比到处手写更稳妥。

## 常见改动怎么做

### 想加一个新的公共数学函数

优先判断它是不是“多个模块都会用到”的基础函数：

- 如果是，放进 `core/`
- 如果只在单个模块内部用，就留在对应 `.cpp` 的匿名命名空间里

### 想换随机采样策略

如果只是某种材质或灯光的采样分布变化：

- 改对应模块，不一定要动 `core/`

如果是底层 RNG 或采样工具统一升级：

- 再考虑改 `global.hpp`

### 想调全局数值稳定参数

主要看：

- `EPSILON`
- 可能还有一些默认常量或工具函数

改这类值时要非常谨慎，因为它会影响整个工程。

## 协作建议

这个模块适合：

- 少量稳定维护
- 少做大幅重构
- 每次修改后快速回归编译和渲染

因为它一旦出错，通常会波及整个代码库。
